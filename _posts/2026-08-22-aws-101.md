**TL;DR:** AWS is Amazon's public cloud platform organized into regions (datacenters), availability zones (isolated facilities within a region), and a shared responsibility model (AWS secures the cloud, you secure what you put in it). Every service you provision — an EC2 instance, an S3 bucket, a VPC — is an API-managed resource with a unique ARN, a region, and a set of IAM permissions. The bugs that bite developers are almost never the service itself; they are overly permissive IAM roles, publicly accessible S3 buckets, and misconfigured security group rules.

## 1. What AWS is (and what it isn't)

AWS is a collection of over 200 cloud services running in Amazon-managed datacenters worldwide. The three categories you will use every day:

- **Compute** — EC2 (VMs), ECS/EKS (containers), Lambda (serverless), Elastic Beanstalk (PaaS)
- **Storage** — S3 (objects), EBS (block volumes for EC2), EFS (NFS filesystem), RDS (relational databases), DynamoDB (NoSQL)
- **Networking** — VPC (virtual network), ELB/ALB (load balancers), Route 53 (DNS), CloudFront (CDN)

Every service is provisioned and managed through the **AWS API** (or CLI, CloudFormation, Terraform), which authenticates every call via IAM, enforces resource-based policies, and tracks every resource in your account.

The mental model: an AWS account is your namespace; every region is an independent set of services; every resource has an ARN like `arn:aws:s3:::my-bucket` or `arn:aws:ec2:us-east-1:123456789012:instance/i-abc123`.

## 2. A real example: a web app with a database

Here is a minimal but realistic deployment: an EC2 instance behind an ALB, backed by RDS PostgreSQL, provisioned via the AWS CLI.

```bash
# Configure AWS credentials
aws configure

# Create a VPC with CIDR 10.0.0.0/16
aws ec2 create-vpc --cidr-block 10.0.0.0/16

# Create a security group allowing HTTP from anywhere
aws ec2 create-security-group --group-name web-sg --description "Web traffic"
aws ec2 authorize-security-group-ingress --group-name web-sg \
  --protocol tcp --port 80 --cidr 0.0.0.0/0

# Launch an EC2 instance (Amazon Linux 2, t2.micro)
aws ec2 run-instances --image-id ami-0abcdef1234567890 --instance-type t2.micro \
  --security-groups web-sg --user-data file://bootstrap.sh

# Create an RDS PostgreSQL database
aws rds create-db-instance --db-instance-identifier myapp-db \
  --db-instance-class db.t3.micro --engine postgres \
  --master-username dbadmin --master-user-password "P@ssw0rd!" \
  --allocated-storage 20
```

## 3. Regions, availability zones, and choosing a datacenter

An **AWS region** is a cluster of datacenters in a geographic area (e.g. `us-east-1` = North Virginia, `eu-west-1` = Ireland). Each region is completely independent — your resources in one region are not visible in another unless you explicitly replicate them.

An **availability zone (AZ)** is one or more discrete datacenters within a region, each with independent power, cooling, and networking. AZs are connected by low-latency fiber. Deploying across multiple AZs is how you achieve high availability — if one AZ fails, traffic shifts to the others.

**Choosing a region**: pick the one closest to your users for lowest latency. Check that every service you need is available in that region — new services launch in a subset of regions first.

## 4. EC2: virtual machines on demand

Amazon EC2 provides resizable virtual machines (instances) in the cloud. You choose an **AMI** (machine image with the OS and pre-installed software), an **instance type** (CPU/memory/storage/network capacity), and a **VPC + subnet** (where the instance lives in the network).

- **t2.micro / t3.micro** — 1 vCPU, 1 GB RAM, burstable CPU credits; free-tier eligible, good for dev/test
- **m5.large** — 2 vCPUs, 8 GB RAM, general-purpose; good for web/app servers
- **c5.2xlarge** — 8 vCPUs, 16 GB RAM, compute-optimized; good for batch processing
- **r5.2xlarge** — 8 vCPUs, 64 GB RAM, memory-optimized; good for databases (not RDS)

The most common EC2 bug is running out of **CPU credits** on a T-series instance — once credits are exhausted, the instance is throttled to a baseline CPU, causing performance degradation until credits replenish.

## 5. S3: object storage at internet scale

Amazon S3 stores unstructured data as **objects** (up to 5 TB each) inside **buckets** (globally unique names). Each object has a URL like `https://{bucket}.s3.{region}.amazonaws.com/{key}` and an access tier:

- **S3 Standard** — frequent reads, low latency, high durability (11 9's)
- **S3 Intelligent-Tiering** — automatic tiering between frequent and infrequent access
- **S3 Glacier** — archival storage, retrieval in minutes to hours

The most common S3 bug is a **publicly accessible bucket** — either a bucket policy that allows `Principal: "*"` or an ACL that grants public read/write. AWS now enables **S3 Block Public Access** by default at the account level, but if you explicitly override it, you can accidentally expose data.

## Review checklist

- [ ] Every AWS resource has a unique ARN and lives in exactly one region
- [ ] Compute choices: EC2 (IaaS), ECS/EKS (containers), Lambda (serverless), Elastic Beanstalk (PaaS)
- [ ] Storage choices: S3 (objects), EBS (block), EFS (NFS), RDS (relational), DynamoDB (NoSQL)
- [ ] IAM permissions and S3 bucket policies are the most common security misconfigurations
- [ ] Deploy across at least two AZs for production workloads
