**TL;DR:** Azure is Microsoft's public cloud platform organized into regions (datacenters), resource groups (logical containers), and control-plane APIs (Azure Resource Manager). Every service you provision — a VM, an App Service plan, a SQL database, a blob container — is an ARM resource with a unique ID, a location, and a set of RBAC permissions. The bugs that bite developers are almost never the service itself; they are region unavailability, throttled API calls, misconfigured network security groups, and accidentally public blob containers.

> **In plain English (30 sec):** Code you already write — Map, function, API call, just bigger.

## 1. What Azure is (and what it isn't)

Azure is a collection of over 200 cloud services that run in Microsoft-managed datacenters around the world. The three categories you will use every day are:

- **Compute** — VMs, App Service, Functions, AKS (Kubernetes), Container Instances
- **Storage** — Blob (objects), Disk (VHDs), Files (SMB), SQL Database, Cosmos DB
- **Networking** — Virtual Network, Load Balancer, Application Gateway, Traffic Manager, DNS

Every service is provisioned and managed through **Azure Resource Manager (ARM)**, the control plane that authenticates all calls (via Azure AD), enforces RBAC permissions, and tracks every resource in a **resource group**.

The mental model: a resource group is a folder; a resource inside it is a file in that folder. Deleting the folder deletes all files. Every resource has a unique ARM ID like `/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.Compute/virtualMachines/{name}`.

## 2. A real example: a web app with a database

Here is a minimal but realistic deployment: an App Service web app backed by Azure SQL Database, provisioned via the Azure CLI.

```bash
# Login and set subscription
az login
az account set --subscription "my-subscription"

# Create a resource group
az group create --name myapp-rg --location eastus

# Create an App Service plan (B1 = basic, 1 core, 1.75 GB RAM)
az appservice plan create --name myapp-plan --resource-group myapp-rg --sku B1

# Create the web app
az webapp create --name myapp-web --resource-group myapp-rg --plan myapp-plan

# Create Azure SQL Server (set a real password in production)
az sql server create --name myapp-sql --resource-group myapp-rg --location eastus \
  --admin-user dbadmin --admin-password "P@ssw0rd!"

# Create a database
az sql db create --server myapp-sql --resource-group myapp-rg --name myapp-db \
  --edition Basic --max-size 2GB

# Configure the web app's connection string
az webapp config connection-string set --resource-group myapp-rg --name myapp-web \
  --settings DefaultConnection="Server=tcp:myapp-sql.database.windows.net;Database=myapp-db;..."
  --connection-string-type SQLAzure
```

## 3. Regions, availability zones, and choosing a datacenter

An **Azure region** is a set of datacenters within a latency-defined perimeter (e.g. "East US", "West Europe"). Each region is paired with another region in the same geography (e.g. East US paired with West US) for geo-redundant storage and planned maintenance.

An **availability zone** is a physically separate datacenter within a region, with independent power, cooling, and networking. Services that support zones (VMs, AKS, Load Balancer) can distribute replicas across zones to survive a single-datacenter failure. Services that are *zonal* (tied to one zone) must be deployed in at least two zones for high availability.

**Choosing a region**: pick the one closest to your users for lowest latency, but check that every service you need is available in that region — not every region supports every SKU.

## 4. App Service: PaaS compute for web apps

Azure App Service is a fully managed PaaS (Platform as a Service) offering that runs your web app inside a sandboxed Windows or Linux worker. You do not manage the OS, the IIS/nginx config, or the security patches — the platform handles those. You only provide the code and a **pricing tier** that determines CPU, memory, and scaling behavior.

- **Free / Shared** — shared worker, capped quota, no custom domain
- **Basic (B1-B3)** — dedicated worker, 1-3 cores, up to 1.75 GB RAM per instance
- **Standard (S1-S3)** — auto-scale up to 10-30 instances, staging slots, traffic manager
- **Premium (P1-P3)** — more memory, private endpoints, faster scaling

The most common App Service bug is hitting the **instance limit** during a traffic spike — the plan does not auto-scale beyond its tier's max instance count.

## 5. Blob storage: objects, not files

Azure Blob Storage stores unstructured data (images, logs, backups) as **blobs** inside **containers** (like S3 buckets). Each blob has a URL like `https://{account}.blob.core.windows.net/{container}/{blob}` and an access tier:

- **Hot** — frequent reads, higher storage cost, no retrieval cost
- **Cool** — infrequent reads, lower storage cost, retrieval cost
- **Archive** — offline, lowest cost, hours to rehydrate

The most common Blob bug is a **public container**: containers default to private access, but if you accidentally set the container ACL to `container` (public read for all blobs) instead of `blob` (public read for known URLs), every blob in that container is visible to anyone with the URL.

## Review checklist

- [ ] Every Azure resource lives in a resource group and has a unique ARM ID
- [ ] Compute choices: App Service (PaaS), VMs (IaaS), Functions (serverless), AKS (containers)
- [ ] Storage choices: Blob (objects), Disk (VHDs), Files (SMB), SQL Database (relational), Cosmos DB (NoSQL)
- [ ] Network security is the most common misconfiguration: NSG rules, public blob containers, open SQL firewalls
